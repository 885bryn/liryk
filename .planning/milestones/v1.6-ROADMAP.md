# Roadmap: Spotify Live Lyrics Web App

## Overview

This milestone adds a toggleable in-fullscreen developer overlay that displays a real-time event log of app activity. The panel must live inside the fullscreen root, must not disrupt lyric layout or live-lock behavior, and must wire to real runtime events across all major operation categories.

## Phase Details

### Phase 21: Panel Container, Toggle, and UX Shell
**Goal**: A developer panel that opens and closes from fullscreen, isolates its scroll from the lyric viewport, displays auth/connection events, auto-scrolls to new entries, and matches the dark fullscreen aesthetic — without touching lyric layout or live-lock behavior.
**Depends on**: Phase 20 (fullscreen viewport ownership model)
**Requirements**: DEV-01, DEV-02, DEV-03, DEV-07, DEV-08, DEV-09
**Success Criteria** (what must be TRUE):
  1. User can tap/click the toggle button while in fullscreen and the developer panel opens; tapping again closes it — and the lyric display is not affected either way.
  2. Scrolling inside the panel log does not disengage live lock on the lyric viewport.
  3. Auth and connection events (token refresh, auth state change, connection status) appear as timestamped entries in the log while the panel is open.
  4. The panel auto-scrolls to the most recent log entry; the user can pause auto-scroll to read an older entry and resume it.
  5. The panel is visually unobtrusive against the dark fullscreen background and does not occlude or shift any lyric rows.
**Plans**: 2 plans

- [ ] 21-02-PLAN.md — Wire toggle, panel JSX, and auth events into FullscreenLyricsPage (Wave 2)

### Phase 22: Event Emission Wiring
**Goal**: The event log populates with real runtime activity — lyrics fetch lifecycle, Spotify API poll results, track changes, playback state updates, and clock drift/reset events — so the developer can observe the full app operation cycle without leaving fullscreen.
**Depends on**: Phase 21
**Requirements**: DEV-04, DEV-05, DEV-06
**Success Criteria** (what must be TRUE):
  1. When lyrics are fetched, the panel shows a timestamped entry indicating whether it was a cache hit, a live fetch (with provider name), or a failure.
  2. When Spotify API polls fire or a track change or playback state update arrives, the panel shows a corresponding timestamped entry.
  3. When the playback clock applies a drift correction or hard reset, the panel shows a timestamped entry with enough detail to distinguish correction type.
  4. All three event categories appear in a single chronological log with no missed events during normal playback — the developer can trace a complete operational sequence without leaving fullscreen.
**Plans**: 1 plan
